## 五、时间规则引擎说明

### 匹配优先级

时间规则分三种类型，按以下优先级匹配：

1. **city 类型**（最高优先级）：精确匹配城市名
   - 例：规则匹配值为"深圳市"，当城市为"深圳市"时精确命中
   
2. **tier 类型**（中等优先级）：按逗号分隔的关键词匹配
   - 例：匹配值为"北京,上海,广州,深圳"，包含关系匹配
   
3. **global 类型**（最低优先级）：全局默认规则，无匹配时使用

### 时间表生成规则

```
输入：城市名（如"深圳市"）
  ↓
查找匹配规则
  ↓
获取：locked_days（锁定天数），max_days（最远天数），
      time_slots（时间段列表），work_days（工作日）
  ↓
从明天起遍历max_days天：
  - 跳过非工作日
  - 前locked_days天：is_full=true，time_slots=[]
  - 之后的天：is_full=false，返回time_slots
```

### 配置示例

| 规则名称 | 类型 | 匹配城市 | 锁定天数 | 效果 |
|---------|------|---------|---------|------|
| 一线城市 | tier | 北京,上海,广州,深圳 | 4 | 前4天约满 |
| 二线城市 | tier | 成都,杭州,武汉,南京 | 3 | 前3天约满 |
| 全局默认 | global | - | 2 | 前2天约满 |

## 六、Puppeteer自动下单流程

### 流程图

```
调用 执行自动下单(订单ID)
         │
         ▼
从账号池获取空闲账号
（is_busy=0, status=1，优先最久未使用）
         │
         ▼
设置账号 is_busy=1
         │
         ▼
启动 Puppeteer 浏览器（无头模式）
         │
         ▼
注入账号Cookie登录京东
         │
         ▼
打开收货地址页面
（https://newaddress.m.jd.com/newAddress/add）
         │
         ▼
自动填写客户信息：
- 收货人姓名
- 手机号
- 省市区
- 详细地址
         │
         ▼
保存收货地址
         │
         ▼
打开京东家政下单页
         │
         ▼
选择预约时间
         │
         ▼
提交订单
    ┌────┴────┐
    │ 失败     │ 成功
    ▼          ▼
重试（最多3次） 获取京东订单号
    │          │
    │          ▼
    │      更新订单状态为已下单
    │      释放账号(is_busy=0)
    ▼
超过重试次数→ 更新状态为失败
释放账号
```

### 注意事项

1. **Cookie维护**：定期手动登录更新Cookie（建议每月一次）
2. **账号池管理**：建议维护3-5个账号，避免单账号操作频繁被封
3. **反爬处理**：已设置移动端User-Agent，减少被识别风险
4. **页面变更**：京东页面结构可能随时变化，需根据实际情况调整选择器

## 九、多备选时间预约功能说明

### 功能介绍
允许客户在预约时选择 1～3 个备用时间，客服按优先级顺序为客户安排，大幅提高预约成功率。

### 功能开关
通过后台管理"系统设置 → 京东家政设置 → 多备选时间设置"控制，默认关闭（`multi_time_enabled: '0'`），开启后不影响历史单选订单。

### 系统设置字段
| 字段 | 默认值 | 说明 |
|------|--------|------|
| multi_time_enabled | 0 | 是否开启（0关闭/1开启） |
| multi_time_max_count | 3 | 客户最多可选几个备选时间 |
| multi_time_tip | 空 | 时间选择页顶部提示文字 |
| multi_time_min_count | 1 | 至少需要选几个才能提交（1=不强制） |

### 数据格式
`visit_times` 字段存储 JSON 数组：
```json
[
  {"date": "2024-01-19", "time": "09:00", "priority": 1},
  {"date": "2024-01-20", "time": "14:00", "priority": 2},
  {"date": "2024-01-21", "time": "10:00", "priority": 3}
]
```

### 向下兼容
原 `visit_date` / `visit_time` 字段保留，存第1优先时间。老数据（无 `visit_times` 字段）在所有界面均正常显示。

### 数据库迁移

### 10.3 签名验证规则

**签名算法（MD5）：**

1. 除 `sign` 字段外，所有参数（包括空值参数）按字段名ASCII码从小到大排序
2. 拼接成 `key1=value1&key2=value2` 格式字符串
3. 前后各追加 `AppSecret + 商户密钥` 的拼接字符串
4. 整个字符串做MD5，转32位大写

**示例：**

假设 AppSecret = `rste57w8rsubsnxsb384ur3u9kn5fzhr`，商户密钥 = `0a091b3aa4324435aab703142518a8f7`

```
密钥串 = rste57w8rsubsnxsb384ur3u9kn5fzhr0a091b3aa4324435aab703142518a8f7
待签名字符串 = {密钥串}{QueryString}{密钥串}
sign = MD5(待签名字符串).toUpperCase()
```

**时间戳验证：** 超过5分钟的请求直接拒绝（返回 code: 408）

### 10.4 卡密加密规则

- 算法：AES/ECB/PKCS7Padding/256位
- 密钥：AppSecret（32字符 = 256位）
- 输出：Base64字符串
- Node.js实现：使用内置 `crypto` 模块，`aes-256-ecb` 模式

**加密数据格式：**

```json
[
  {
    "cardNo": "卡密code",
    "cardPwd": "",
    "expireTime": "2024-12-31 23:59:59"
  }
]
```

### 10.5 商品编号（productNo）规则

本系统的商品编号格式：`{business_type}_{service_type}_{service_hours}h`

阿奇所平台调用本系统接口时，使用以下签名算法：

```javascript
// 1. 将所有请求参数（不含sign字段）按 key 的 ASCII 码升序排列
// 2. 拼接为 key=value&... 形式的字符串
// 3. 末尾拼接 &key={商户密钥}
// 4. 进行 MD5 加密（转大写）

const 待签名字符串 = Object.keys(参数).sort()
  .filter(k => k !== 'sign' && 参数[k] !== undefined && 参数[k] !== '')
  .map(k => `${k}=${参数[k]}`)
  .join('&') + `&key=${商户密钥}`;
const sign = md5(待签名字符串).toUpperCase();
```

### 卡密AES加密

下单接口返回的 `cards` 字段使用 AES-ECB 加密：

```javascript
// 加密：AES-ECB 模式，PKCS7 填充，密钥为 AppSecret 的前16位
const CryptoJS = require('crypto-js');
const key = CryptoJS.enc.Utf8.parse(appSecret.substring(0, 16));
const encrypted = CryptoJS.AES.encrypt(JSON.stringify(cardsArray), key, {
  mode: CryptoJS.mode.ECB,
  padding: CryptoJS.pad.Pkcs7,
});
const result = encrypted.toString(); // Base64字符串
```

### 撤单退款流程

```
阿奇所发起撤单请求
  ↓
POST /agisoAcprSupplierApi/order/cancelOrder { orderNo: "xxx" }
  ↓
查找 Card 表中 agiso_order_no = orderNo
  ↓
幂等判断：sup_status=2 → 直接返回 cancelStatus:20
  ↓
按 business_type 分业务处理：

家政(jiazheng)：
  - 查 orders 表（status != 4）
  - status=2（已下单）→ 拒绝撤单（cancelStatus:30）
  - status=其他 → 取消订单（status=4），写日志
  - 卡密 status=2（已失效），sup_status=2（已撤单）

洗衣(xiyifu)：
  - 如有 laundry_order_id → 调用鲸蚁 sync-status（status=-1）
  - 取消订单（status=4），写日志
  - 卡密 status=2（已失效），sup_status=2（已撤单）

充值(topup)：
  - 无论订单状态，均允许撤单
  - 取消订单（status=4），写日志
  - 卡密 status=2（已失效），sup_status=2（已撤单）

  ↓
返回 { code: 200, data: { cancelStatus: 20 } }
```

## 十八、鲸蚁生活洗衣对接说明

### 鉴权流程

```
1. 调用 POST {api_url}/api/oauth/token { app_id, app_secret }
2. 获取 access_token 和 tenantId
3. 后续请求 Header 中携带：
   - out-token: {access_token}
   - tenant-id: {tenantId}
4. Token 有效期约2小时，提前5分钟自动刷新
5. Token 持久化到 Setting 表，服务重启后无需重新登录
```

## 十九、统一卡密中心使用说明

统一卡密中心（`/admin/card-center`）整合了三种业务类型的卡密管理，避免在多个页面间来回切换。

### 主要功能

**统一生成 Tab：**
1. 选择业务类型（家政/洗衣/充值）
2. 可选关联SUP商品（自动填充字段）
3. 根据业务类型填写对应字段（家政/洗衣/充值配置不同）
4. 填写生成数量（1-1000）、备注、过期时间
5. 点击生成 → 显示生成结果，支持复制完整链接或仅卡密

**统一列表 Tab：**
- 支持按业务类型、状态、关键词筛选
- 一张表格显示三种业务的所有卡密
- 直接显示完整链接，支持快速复制
- 支持作废/删除操作

**批次管理 Tab：**
- 支持按业务类型筛选
- 显示各批次统计（总数/已用/未用）
- 点击"查看卡密"可弹窗查看批次内所有卡密及链接
- 支持复制完整链接、复制仅卡密、删除批次

### 链接前缀规则

| 业务类型 | 链接格式 |
|----------|---------|
| 家政 | `{site_url}/{code}` |
| 洗衣 | `{site_url}/xi/{code}` |
| 充值 | `{site_url}/cz/{code}` |

`site_url` 从系统设置中读取。
