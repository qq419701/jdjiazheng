# 京东代下单管理系统 - 技术文档 v3.0

## 一、系统概览

本系统为**三业务合一**的卡密代下单管理平台，支持家政、洗衣、充值三类业务的卡密分发、订单管理和后台运营。

### 三大业务

| 业务 | 前端路径 | 前端目录 | 核心流程 |
|------|----------|----------|----------|
| 🏠 家政 | `/{site}/jz/{卡密}` | `frontend-h5/` | 用户填写预约信息 → 系统Puppeteer自动在京东下单 |
| 🧺 洗衣 | `/{site}/xi/{卡密}` | `frontend-xi/` | 用户填写取/收件地址 → 自动推送鲸蚁生活系统 |
| 💳 充值 | `/{site}/cz/{卡密}` | `frontend-cz/` | 用户填写充值账号 → 人工充值 → 管理员确认完成 |

## 八、Puppeteer自动下单（家政）

通过京东账号池自动在京东平台下单家政服务。
- 支持账号池轮询（`is_busy` 状态控制并发）
- 失败自动重试3次
- 支持Cookie注入登录

## 九、鲸蚁生活API对接（洗衣）

洗衣订单提交后自动推送到鲸蚁生活洗护系统，支持：
- 自动获取Token / 使用固定 out-token
- 物流轨迹查询
- 状态回调接收

配置路径：业务设置 → 洗衣设置 → 鲸蚁API配置

### 回调机制说明

鲸蚁采用**主动回调**方式通知订单状态变化，不支持主动查询。

- 回调时间：推送订单后，鲸蚁师傅接单时（一般30分钟～数小时），如服务区不覆盖则无回调
- 回调地址：`{site_url}/api/laundry/callback`，需在鲸蚁后台「应用设置」中配置
- 回调字段 `waybillCode`：取件快递单号，在 status=1（已分配）或 status=2（已取件）时携带
- 状态流转：1=已分配 → 2=已取件 → 3=已入厂 → 4=预检中 → 5=已回寄 → 6=已送达 → 10=完成

## 十四、业务模块开关（.env 全链路控制）

### 功能说明

通过修改服务器 `backend/.env` 文件，可以独立开启或关闭某个业务模块。关闭后效果：

| 范围 | 效果 |
|------|------|
| H5 卡密验证接口 | 返回"业务未开放"，用户无法使用卡密 |
| H5 订单提交接口 | 返回"业务未开放"，无法下单 |
| 后台订单中心 | 对应业务 Tab 消失 |
| 后台卡密工作台 | 对应业务分组消失 |
| 后台顶部工具栏 | 预览前端/卡密作废/导出CSV 对应业务选项消失 |

### 配置方式

1. SSH 登录服务器
2. 编辑 `backend/.env` 文件：
   ```bash
   vim /www/wwwroot/<项目目录>/backend/.env
   ```
3. 找到或追加以下配置：
   ```env
   # 🏠 家政业务（true=开启 / false=关闭）
   BUSINESS_JIAZHENG_ENABLED=true
   # 🧺 洗衣业务（true=开启 / false=关闭）
   BUSINESS_XIYIFU_ENABLED=true
   # 💳 充值业务（true=开启 / false=关闭）
   BUSINESS_TOPUP_ENABLED=true
   # ⚔️ 三角洲业务（true=开启 / false=关闭）
   BUSINESS_SJZ_ENABLED=true
   ```
4. 重启后端生效：
   ```bash
   pm2 restart <进程名>
   ```

### 实现原理

```
backend/.env
  ↓ dotenv 读取
backend/config/business.js（业务开关模块）
  ↓ require 引入
backend/routes/api.js（H5接口层拦截）
  ↓ GET /admin/api/settings 透传
frontend-admin/stores/module.js（前端 Pinia Store）
  ↓ 响应式提供给各组件
OrderCenter.vue / CardWorkbench.vue（动态过滤 Tab 和业务列表）
```

> ⚠️ `.env` 文件不进入 git 版本控制（已在 `.gitignore` 中），对外完全透明，其他人看不到开关状态。

## 十五、update.sh 脚本说明（v5.0）

### 变更内容（v4.2 → v5.0）

| 项目 | v4.2 | v5.0 |
|------|------|------|
| `PROJECT_DIR` | 写死 `/www/wwwroot/jdjiazheng` | 自动检测脚本所在目录 |
| `PM2_APP_NAME` | 写死 `jdjiazheng` | 自动取目录名 |
| `GITHUB_REPO` | 写死 `qq419701/jdjiazheng` | 自动取目录名拼接 |
| 多系统支持 | ❌ 新旧系统会互相影响 | ✅ 每个系统独立，互不干扰 |

### 原理说明

```bash
# 自动检测项目目录（脚本放在哪里，就操作哪里）
PROJECT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
# 自动取目录名作为 PM2 进程名
PM2_APP_NAME="$(basename "$PROJECT_DIR")"
```

这样同一台服务器部署多个项目时（如旧系统 `jdjiazheng`、新系统 `yuyuexitong`），各自的 `update.sh` 只会操作自己的目录，永远不会混淆。

### 使用方式不变

```bash
./update.sh           # 全量更新
./update.sh backend   # 仅后端
./update.sh sjz       # 仅三角洲前端
# ...其他命令同 v4.2
```
